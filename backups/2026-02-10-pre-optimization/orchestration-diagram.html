Full Lifecycle (Linear View)

1. PLAN → 2. EXECUTE → 3. GEMINI REVIEW → 4. FIX ↺ → 5. CLAUDE REVIEW → 6. FIX ↺ → 7. SHIP
Key: Phases 3-4 loop until all Gemini reviewers give A+. Then phases 5-6 loop until all Claude Opus reviewers give A+. Only then does Phase 7 (Ship) happen.

All Rules & Protocols

Conductor Protocol (ABSOLUTE)

Conductor DOES

  • Orchestrate agents (spawn, track, summarize)
  • Git operations (commit, push, branch, PR via gh)
  • Update status files (PROGRESS.md, PROJECT_STATUS.md)
  • Update memory files (MEMORY.md, *.md)
  • Communicate with user (tables, summaries)
  • Quick file checks (Glob, Grep patterns)

Conductor NEVER

  • Write/edit code files (.ts, .tsx, .py, .js, .css)
  • Read large source files (>50 lines)
  • Run tests or builds directly
  • Paste code blocks in responses
  • Debug failing tests
  • Combine 8 reviewers into 1 agent

Team Rules

Team-First Rule

  • Default to TeamCreate for ANY real work
  • Fire-and-forget Tasks ONLY for single quick lookups
  • Dev Team Leader does NOT write code
  • Executors have file ownership boundaries
  • Teams let agents coordinate + message when blocked

Dev Team Leader Rules

  • Spawned via TeamCreate (not fire-and-forget)
  • Orchestrates executor agents — never writes code
  • Splits work with file ownership boundaries
  • CAN delegate to Gemini where appropriate
  • Reports summary back to Conductor

Review Rules

Gemini Review Rules (Stage 1)

  • Gemini Review Coordinator = dedicated team member
  • Spawns SEPARATE Gemini instance per reviewer role
  • ALWAYS send FULL code files + diff (never just diff)
  • Skip only clearly irrelevant roles
  • Iterate until ALL relevant reviewers give A+
  • Use subagents to parallelize if faster

Claude Review Rules (Stage 2)

  • Separate Claude Opus agent per reviewer role
  • Each agent has its OWN independent context
  • Agents read files themselves (tool access)
  • All agents run in parallel
  • Iterate until ALL reviewers give A+
  • Never combine reviewers into one agent

Fix Flow Rules

Fix Thresholds

  • 3+ issues: Dev Team Leader → executor team with file ownership
  • 1-2 issues: Single executor agent
  • NEVER spawn single agent for 3+ fixes
  • After fixes: re-review ONLY failed roles
  • Use same stage (Gemini or Claude) that found issues

Planning Rules

  • Separate planning agent per perspective
  • ASK user which roles each time (default: all 8)
  • Each agent analyzes from their domain
  • Conductor aggregates into unified plan
  • User approves before execution begins

Gemini Delegation Rules

Gemini A-Tier (Good At)

  • Code review (with full code context)
  • Codebase analysis (1M context window)
  • JSON validation, CSS/UI review
  • Documentation writing
  • PR descriptions

Gemini D-Tier (Never Use For)

  • Auth/payment implementation
  • Git operations
  • Multi-step agentic tasks
  • Debugging complex issues
  • Tasks requiring tool access
CRITICAL: MCP is BROKEN on Windows. NEVER use MCP tools for Gemini. ONLY use Bash: gemini -p "prompt" --model gemini-3-pro-preview 2>&1 with run_in_background: true.

Context Hygiene Rules

Context Protection

  • Summarize agent output in 2-3 lines max
  • Never paste >100 lines of tool output
  • Batch work into agents (1 agent with 5 steps > 5 tool calls)
  • Background everything long-running
  • Memory over context — write decisions to memory files

Agent Prompt Packaging

  • Include file PATHS in agent prompts (agents read files themselves)
  • Never read files yourself then pass content to agents
  • Let agents gather their own context
  • Describe what to do, not how — agents are smart

Git & Session Rules

Session Start (MANDATORY)

  • git pull origin main
  • Check for open PRs: gh pr list --state open
  • Read PROGRESS.md for current status
  • Read PROJECT_STATUS.md for full state
  • Present status summary to user

Session End (MANDATORY)

  • Update PROGRESS.md with what was done
  • Update PROJECT_STATUS.md if state changed
  • Push both to main
  • Never leave user guessing where we left off

Git Rules

  • Never push code to main (only status files)
  • Rewrite branches: rewrite/phase-N-description
  • Squash merge PRs for clean history
  • Conventional commit messages
  • gh path: "C:\Program Files\GitHub CLI\gh.exe"

PR Format (MANDATORY)

  • Summary table: Item | Severity | File(s) | Status
  • Review grades table: Reviewer | Gemini R1 | R2 | Claude | Fixes
  • Test plan with verification commands
  • Multi-agent status table at end of orchestration

Red Flags (You're Doing It Wrong If...)

  • Reading a .ts/.tsx/.py file → STOP, delegate to explore agent
  • Writing code in Edit/Write → STOP, delegate to executor agent
  • Tool output >100 lines → STOP, you should have delegated
  • Running vitest/pytest directly → STOP, delegate to agent
  • Combining 8 reviewers into 1 agent → STOP, each needs own context
  • Sending Gemini only a diff → STOP, include full file contents
  • Single agent fixing 3+ issues → STOP, use Dev Team Leader + team
  • Dev Team Leader writing code → STOP, should delegate to executors
  • Context filling fast → STOP, you're doing too much yourself
  • Spawning 3+ fire-and-forget Tasks → STOP, use TeamCreate
