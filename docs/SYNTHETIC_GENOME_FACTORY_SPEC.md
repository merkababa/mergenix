## 1. Overview

The Synthetic Test Genome Factory generates realistic raw DNA data files in all formats supported by the Mergenix parser (`packages/genetics-engine/src/parser.ts`). It produces files that:

- Parse correctly through `parseGeneticFile()` and all format-specific parsers
- Pass all format-specific validators (`validate23andMe`, `validateAncestryDNA`, `validateMyHeritage`, `validateVcf`)
- Contain deterministic, reproducible genotype data for test assertions
- Can inject specific pathogenic variants for carrier/trait/PGx/PRS testing
- Cover all edge cases the parser must handle (or reject gracefully)

The factory will live at `packages/genetics-engine/src/test-utils/genome-factory.ts` and be importable by any test file in the monorepo.

### Relationship to Existing Code

The factory must generate files compatible with:

| Module | File | What it consumes |
|--------|------|------------------|
| Parser | `parser.ts` | Raw file content (string) |
| Carrier | `carrier.ts` | `GenotypeMap` (rsid -> genotype) |
| Traits | `traits.ts` | `GenotypeMap` |
| PGx | `pgx.ts` | `GenotypeMap` |
| PRS | `prs.ts` | `GenotypeMap` |
| Ethnicity | `ethnicity.ts` | `GenotypeMap` + population |

The factory generates raw file content (string). Tests that need a `GenotypeMap` directly can use a separate helper `buildGenotypeMap()` that bypasses file generation.

## 2. File Format Specifications

### 2.1 23andMe Format

The Mergenix parser detects 23andMe files by:
1. Comment lines containing "23andme" (case-insensitive)
2. Comment lines containing "rsid" + "chromosome" or "position"
3. Heuristic: 4-column tab-separated data with `rs`/`i` prefixed IDs and short genotype strings

**Column layout:** Tab-separated, 4 columns

```
rsid	chromosome	position	genotype
```

**Genotype encoding:** Combined alleles in a single field (e.g., `AA`, `AG`, `--` for no-call)

**Valid rsID prefixes:** `rs` (dbSNP), `i` (internal/indel)

**Valid chromosome values:** `1`-`22`, `X`, `Y`, `MT`, `M`

### 2.2 AncestryDNA Format

Detected by:
1. Comment lines containing "ancestrydna" (case-insensitive)
2. 5-column tab-separated header with "allele1" and "allele2"

**Column layout:** Tab-separated, 5 columns

```
rsid	chromosome	position	allele1	allele2
```

**Genotype encoding:** Separate allele columns. Parser concatenates `allele1 + allele2`. No-call uses `0` for allele values.

**Valid rsID prefixes:** `rs`, `i`

**Valid chromosome values:** `1`-`22`, `X`, `Y`, `MT`, `23` (=X), `24` (=Y), `25` (=MT/PAR)

#### 2.2.1 AncestryDNA v1

```
#AncestryDNA raw data download
#This file was generated by AncestryDNA https://dna.ancestry.com
#This raw data file was generated at: 07/15/2014 09:23:45 UTC
#Data was collected using AncestryDNA array version: V1.0
#Data is formatted using AncestryDNA converter version: V1.0
#Below is a text version of your data. Fields are TAB-separated.
#Each line corresponds to a single SNP. For each SNP, we provide its identifier (an rsid or an
#internal id), its location on a reference human genome, and the two alleles observed at this
#position. Our genotyping methodology determines one allele per strand (forward and reverse).
rsid	chromosome	position	allele1	allele2
rs4477212	1	82154	A	A
rs3094315	1	752566	A	G
rs3131972	1	752721	G	G
rs12124819	1	776546	A	A
rs11240777	1	798959	A	G
rs6681049	1	800007	C	C
rs4970383	1	838555	A	C
rs4475691	1	846808	C	T
rs7537756	1	854250	A	G
```

Key characteristics:
- Uses numeric chromosomes for sex chromosomes: `23` = X, `24` = Y, `25` = MT/PAR
- Approximately **700,000** SNPs
- Header mentions "V1.0"

### 2.3 MyHeritage/FTDNA Format

Detected by:
1. CSV header row with `RSID`, `CHROMOSOME`, `POSITION`, `RESULT`
2. Heuristic: 4-column comma-separated data with `rs`/`i`/`VG` prefixed IDs

**Column layout:** Comma-separated, 4 columns (fields may or may not be quoted)

```
RSID,CHROMOSOME,POSITION,RESULT
```

**Genotype encoding:** Combined alleles in RESULT column (e.g., `AA`, `AG`, `--` for no-call)

**Valid rsID prefixes:** `rs`, `i`, `VG` (proprietary MyHeritage IDs)

**Valid chromosome values:** `1`-`22`, `X`, `Y`, `MT`

**No comment lines.** This is the only format without `#` comment lines.

#### Example (Unquoted)

```
RSID,CHROMOSOME,POSITION,RESULT
rs4477212,1,82154,AA
rs3094315,1,752566,AG
rs3131972,1,752721,GG
rs12124819,1,776546,AA
rs11240777,1,798959,AG
rs6681049,1,800007,CC
VG00001,1,999999,CT
i713426,1,817186,CC
rs4970383,1,838555,AC
rs4475691,1,846808,CT
```

#### Example (Quoted -- MyHeritage variation)

```
"RSID","CHROMOSOME","POSITION","RESULT"
"rs4477212","1","82154","AA"
"rs3094315","1","752566","AG"
"rs3131972","1","752721","GG"
"rs12124819","1","776546","AA"
"rs11240777","1","798959","AG"
"rs6681049","1","800007","CC"
"VG00001","1","999999","CT"
"i713426","1","817186","CC"
"rs4970383","1","838555","AC"
```

The parser uses `parseCsvLine()` which handles both quoted and unquoted fields per RFC 4180.

### 2.4 VCF Format

Detected by:
1. `##fileformat=VCF` meta-information line
2. `##` comments with `#CHROM` header line

**Column layout:** Tab-separated, 10+ columns (mandatory 8 + FORMAT + 1+ SAMPLE columns)

```
#CHROM	POS	ID	REF	ALT	QUAL	FILTER	INFO	FORMAT	SAMPLE_NAME
```

**Genotype encoding:** GT field in FORMAT/SAMPLE columns. Alleles encoded as integer indices (`0` = REF, `1` = first ALT, `2` = second ALT). Separator: `/` (unphased) or `|` (phased).

**Parser behavior:**
- Only extracts variants with `rs` prefixed IDs (skips `.` and other IDs)
- Only extracts SNPs (single-base REF and all single-base ALT alleles)
- Skips indels (multi-base REF or ALT)
- Skips no-calls (`./.`, `.|.`, `.`)
- Skips half-calls (e.g., `0/.`)
- Only uses the first sample column (index 9)
- Skips lines where FORMAT does not contain `GT`

- VCF 4.2 adds `AN` (Total Alleles) in INFO

## 3. Generation Parameters (Factory API)

### 3.1 Core Parameters

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `format` | `FileFormat \| '23andme-v3' \| '23andme-v4' \| '23andme-v5' \| 'ancestrydna-v1' \| 'ancestrydna-v2' \| 'vcf-4.1' \| 'vcf-4.2'` | `'23andme'` | Target output format. Plain `'23andme'` defaults to v5 header. |
| `variantCount` | `number` | `1000` | Total number of variant lines to generate (excluding headers/comments). For golden files use exact panel coverage; for stress tests go up to 700,000+. |
| `seed` | `number` | `42` | PRNG seed for deterministic reproducibility. Same seed + same params = identical output. |
| `includeNoCallRate` | `number` | `0.02` | Fraction of variants to mark as no-call (`--` or `0` or `./.`). Range: 0.0 to 1.0. |

### 3.2 Mutation Injection

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `mutations` | `MutationInjection[]` | `[]` | Array of specific genotypes to force at specific rsIDs. These override any randomly generated genotype. |

```typescript
interface MutationInjection {
  /** rsID to inject (e.g., "rs75030207") */
  rsid: string;
  /** Exact genotype to set (e.g., "CT" for het carrier, "TT" for affected) */
  genotype: string;
  /** Chromosome where this rsID resides (e.g., "7" for CFTR) */
  chromosome: string;
  /** Genomic position (GRCh37/hg19 coordinates) */
  position: number;
}
```

### 3.3 Carrier Scenario Presets

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `carrierScenarios` | `CarrierScenario[]` | `[]` | High-level carrier scenarios to inject. Each resolves to one or more `MutationInjection` entries internally. |

```typescript
type CarrierScenarioType =
  | 'heterozygous_carrier'   // One pathogenic + one reference allele
  | 'affected_homozygous'    // Two pathogenic alleles (AR)
  | 'compound_het'           // Two different pathogenic alleles at same gene (requires 2 rsIDs)
  | 'x_linked_carrier_female' // Female: one pathogenic on X (heterozygous)
  | 'x_linked_hemizygous_male' // Male: single pathogenic allele on X
  | 'not_tested';            // rsID absent from file (simulates chip gap)

interface CarrierScenario {
  /** Which scenario to apply */
  type: CarrierScenarioType;
  /** rsID(s) involved. Compound het requires exactly 2 rsIDs in the same gene. */
  rsids: string[];
  /**
   * If not provided, the factory looks up the rsID in carrier-panel.json
   * to determine pathogenic_allele, reference_allele, chromosome, and position.
   */
  panelLookup?: boolean;
}
```

**Resolution rules for each scenario:**

| Scenario | Genotype set at rsID | Notes |
|----------|---------------------|-------|
| `heterozygous_carrier` | `ref + pathogenic` (e.g., `CT`) | Standard het carrier |
| `affected_homozygous` | `pathogenic + pathogenic` (e.g., `TT`) | Homozygous affected |
| `compound_het` | rsID[0]: `ref + pathogenic`, rsID[1]: `ref + pathogenic` | Both must be in the same gene |
| `x_linked_carrier_female` | `ref + pathogenic` | Chromosome must be X |
| `x_linked_hemizygous_male` | Single `pathogenic` allele | For 23andMe/MyHeritage: single char. For AncestryDNA: `pathogenic` + `0`. For VCF: haploid GT `1`. |
| `not_tested` | rsID omitted from output entirely | Simulates SNP not on chip |

### 3.4 Population / Ancestry

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `population` | `Population` | `'Global'` | Population for allele frequency-weighted random genotype generation. Uses `ethnicity-frequencies.json` data. |

```typescript
type Population =
  | 'African/African American'
  | 'East Asian'
  | 'South Asian'
  | 'European (Non-Finnish)'
  | 'Finnish'
  | 'Latino/Admixed American'
  | 'Ashkenazi Jewish'
  | 'Middle Eastern'
  | 'Global';
```

When a population is specified, non-injected carrier panel SNPs should use population-specific allele frequencies from `ethnicity-frequencies.json` to determine genotype probabilities. For SNPs not in the ethnicity frequency database, use Hardy-Weinberg equilibrium with a default minor allele frequency of 0.15.

### 3.5 Sex Specification

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `sex` | `'male' \| 'female'` | `'female'` | Biological sex. Affects X/Y chromosome genotype encoding. Males get hemizygous (single allele) X chromosome calls and Y chromosome calls. Females get diploid X calls and no Y calls. |

### 3.6 Output Modifiers

| Parameter | Type | Default | Description |
|-----------|------|---------|-------------|
| `lineEnding` | `'\n' \| '\r\n' \| '\r'` | `'\n'` | Line ending character(s). |
| `includeBOM` | `boolean` | `false` | Prepend UTF-8 BOM (`\uFEFF`) to output. |
| `quoteStyle` | `'none' \| 'all' \| 'mixed'` | `'none'` | For MyHeritage format only: quote all fields, no fields, or randomly quote some. |
| `corruptionMode` | `CorruptionMode \| null` | `null` | Apply a specific corruption for edge case testing. See Section 4. |

## 4. Edge Case Test Files

The factory must support generating files with specific defects for negative/robustness testing. Each edge case is activated via the `corruptionMode` parameter or a dedicated generator function.

### 4.1 Corruption Modes

```typescript
type CorruptionMode =
  | 'bom'                    // UTF-8 BOM at file start
  | 'mixed_line_endings'     // Mix of \r\n, \n, \r within one file
  | 'duplicate_rsids'        // Same rsID appears 2+ times with different genotypes
  | 'mid_file_comments'      // Comment lines (#) interspersed in data section
  | 'very_long_lines'        // Some data lines are 10,000+ characters
  | 'unicode_header'         // Unicode characters in header/comment lines
  | 'empty_genotypes'        // Genotype fields that are empty string (not --)
  | 'multi_allelic'          // VCF-only: sites with 3+ ALT alleles
  | 'indels'                 // VCF-only: insertion/deletion variants
  | 'malformed_quality'      // VCF-only: non-numeric QUAL field, or "."
  | 'truncated'              // File ends mid-line (no trailing newline, incomplete last field)
  | 'zero_valid_variants'    // File has headers but every data line is malformed/no-call
  | 'very_large'             // 700,000+ variant lines (streaming parser stress test)
  | 'extra_columns'          // Data lines have more columns than expected
  | 'missing_columns'        // Some data lines have fewer columns than expected
  | 'blank_lines'            // Random blank lines scattered throughout
  | 'windows_crlf'           // All lines use \r\n
  | 'tab_in_csv'             // MyHeritage: tab characters within CSV fields
  | 'missing_header'         // File starts directly with data, no header/comments
  | 'phased_genotypes';      // VCF: all GT use | separator instead of /
```

### 4.2 Edge Case File Specifications

Each edge case file should be generatable by calling:

```typescript
generateEdgeCaseFile(caseId: EdgeCaseId): string
```

#### 4.2.1 BOM File

- Prepends `\uFEFF` (U+FEFF, UTF-8 BOM: bytes `EF BB BF`) before the first character
- The parser's `iterateLines` uses `indexOf('\n')` and `trimEnd()` -- the BOM will appear as the first character of the first line
- **Expected parser behavior:** The BOM becomes part of the first comment line (`# ...`). Since comment lines are skipped, this should NOT affect parsing for commented formats (23andMe, AncestryDNA, VCF). For MyHeritage (no comments), the BOM prefixes the header: `\uFEFFRSID,CHROMOSOME,...`. The `parseCsvLine` will include the BOM in the first field name, which may cause the header skip check (`rsid.toLowerCase() === 'rsid'`) to FAIL because `\uFEFFrsid !== 'rsid'`.
- **This is a known parser gap.** The factory must generate this file so that a BOM-handling fix can be tested.

#### 4.2.2 Mixed Line Endings

- Uses `\n` for first 30% of lines, `\r\n` for next 40%, `\r` for remaining 30%
- **Expected parser behavior:** `iterateLines()` splits on `\n` and calls `trimEnd()` which strips `\r`. Lines ending in `\r` only (no `\n`) will NOT be split by `indexOf('\n')` -- they will be concatenated with the next line. **This is a known parser gap for bare `\r`.**

#### 4.2.3 Duplicate rsIDs

- 10% of rsIDs appear twice with different genotypes (e.g., first `rs4477212 -> AA`, later `rs4477212 -> AG`)
- **Expected parser behavior:** The parser uses `snps[rsid] = genotype` which means the LAST occurrence wins (object property overwrite). This is correct behavior for deduplication.

#### 4.2.4 Mid-File Comments

- 5 comment lines (`# Unexpected comment`) inserted at random positions within the data section
- **Expected parser behavior:** Should be skipped by the `startsWith('#')` check. Only relevant for 23andMe, AncestryDNA, and VCF. MyHeritage has no comment handling -- the `parseCsvLine` would parse `#` lines as malformed data and skip them (< 4 fields or invalid rsID prefix).

#### 4.2.5 Very Long Lines

- 3 data lines padded with trailing whitespace to 10,000+ characters
- **Expected parser behavior:** `trimEnd()` strips trailing whitespace. `split('\t')` or `parseCsvLine` handles the actual data fields normally. Should parse correctly.

#### 4.2.6 Unicode Header

- Comment/header lines include Unicode characters: `# Generated by 23andMe\u2122 \u00A9 2024`
- **Expected parser behavior:** Parser skips comment lines entirely, so this should be harmless.

#### 4.2.7 Empty Genotypes

- 5% of data lines have empty genotype field (empty string, not `--` or `0`)
- **Expected parser behavior:** Parser checks `genotype === '--' || genotype === ''` and skips both. Should work correctly.

#### 4.2.8 Multi-Allelic Sites (VCF only)

- 3% of VCF lines have 3+ ALT alleles (e.g., `A	G,C,T`)
- GT values like `1/2`, `0/3`, `2/3`
- **Expected parser behavior:** Parser splits ALT on `,`, checks all are single-char. Multi-allelic SNPs with single-char alleles should parse correctly. The test should verify `1/2` maps to the correct alleles.

#### 4.2.9 Indels (VCF only)

- 5% of VCF lines have multi-base REF or ALT (e.g., `ATG -> A`, `A -> ATCG`)
- **Expected parser behavior:** Skipped by `ref.length !== 1` check or `altAlleles.some(a => a.length !== 1)` check.

#### 4.2.10 Malformed Quality Scores (VCF only)

- QUAL field set to `.` (missing), `Inf`, `-1`, or non-numeric strings like `HIGH`
- **Expected parser behavior:** Parser does not inspect QUAL at all -- it only reads columns 2 (ID), 3 (REF), 4 (ALT), 8 (FORMAT), 9 (SAMPLE). Should parse correctly regardless of QUAL value.

#### 4.2.11 Truncated File

- File ends mid-line: last line is `rs4477212\t1\t821` (no genotype column, no newline)
- **Expected parser behavior:** `iterateLines()` yields the final partial line. `split('\t')` produces fewer than 4 columns (or 5 for AncestryDNA), so the line is skipped. Remaining valid lines should parse correctly.

#### 4.2.12 Zero Valid Variants

- File has correct headers but every data line is either:
  - No-call (`--`, `0`, `./.`)
  - Malformed (wrong column count)
  - Invalid rsID prefix
- **Expected parser behavior:** All parsers throw `'No valid SNP data found'` error when `snpCount === 0`.

#### 4.2.13 Very Large File

- 700,000+ variant lines (realistic 23andMe v5 file size)
- Approximately 25-35 MB of text
- Must be deterministic from seed
- Used for streaming parser performance testing and `MAX_SNP_COUNT` boundary testing
- **Expected parser behavior:** Should parse successfully. The `checkSnpLimit()` threshold is 10,000,000 so 700K is well within bounds.

#### 4.2.14 Phased Genotypes (VCF only)

- All GT fields use `|` separator: `0|1`, `1|0`, `1|1`
- **Expected parser behavior:** Parser detects `|` via `gtValue.includes('|')` and splits correctly. `1|0` maps to different allele order than `0|1` (which is correct for phased data).

## 8. SNP Universe and Allele Frequency Data

### 8.1 SNP Sources

The factory draws rsIDs and allele information from these sources (in priority order):

1. **Carrier panel** (`carrier-panel.json`): 2,715 entries with `rsid`, `pathogenic_allele`, `reference_allele`, `inheritance`, chromosome (inferred from gene)
2. **Trait SNPs** (`trait-snps.json`): 79 entries with `rsid`, `alleles.ref`, `alleles.alt`, `chromosome`
3. **PGx panel** (`pgx-panel.json`): ~12 genes, each with `defining_snps` array of rsIDs
4. **PRS weights** (`prs-weights.json`): ~10 conditions, each with ~20-35 SNPs including `rsid`, `effect_allele`, `chromosome`
5. **Filler SNPs**: For padding to reach `variantCount`, use a built-in list of common SNPs with realistic positions

### 8.2 Chromosome-Position Mapping

The factory needs a mapping of rsID -> (chromosome, position) for file generation. Sources:

- Carrier panel: Does NOT include chromosome/position. Must be hard-coded in a lookup table within the factory. The spec provides positions for all SNPs used in golden files (Section 5).
- Trait SNPs: Include `chromosome` but NOT position. Positions must be hard-coded.
- PGx panel: Includes `chromosome` per gene. Positions must be hard-coded per defining SNP.
- PRS weights: Include `chromosome`. Positions must be hard-coded.

### 8.3 Allele Frequency Generation

For non-injected variants, the factory assigns genotypes using Hardy-Weinberg equilibrium:

Given minor allele frequency `p`:
- P(homozygous reference) = `(1-p)^2`
- P(heterozygous) = `2 * p * (1-p)`
- P(homozygous alternate) = `p^2`

The PRNG (seeded) generates a uniform random number [0,1) and assigns genotype based on these thresholds.

For carrier panel SNPs with ethnicity frequency data:
- Use the population-specific frequency from `ethnicity-frequencies.json`
- E.g., for `rs334` (Sickle Cell) in `'African/African American'`: frequency = 0.083

For SNPs without frequency data: use default MAF = 0.15.

### 8.4 PRNG Requirements

- Must use a **seedable** PRNG (NOT `Math.random()`)
- Recommend: Mulberry32 or similar simple 32-bit PRNG
- Same seed + same parameters must produce byte-identical output
- The PRNG implementation should be included in the factory module (no external dependency)

The factory must have ZERO external dependencies. It uses only:

- Internal PRNG

### 9.3 Performance Constraints

- `generateSyntheticGenome()` with `variantCount: 1000` must complete in < 50ms
- `generateSyntheticGenome()` with `variantCount: 700000` must complete in < 5 seconds
- String concatenation should use array join, not repeated `+=`

### 9.4 Testing the Factory Itself

The factory needs its own test suite (`tests/genome-factory.test.ts`) verifying:

1. **Determinism:** Same seed produces identical output across calls
2. **Format correctness:** Generated files pass `detectFormat()` for the requested format
3. **Parse round-trip:** `parseGeneticFile(content)` returns genotypes matching `expectedGenotypes`
4. **Mutation injection:** Injected genotypes appear in both the raw content and parsed output
5. **Carrier scenario resolution:** Each `CarrierScenarioType` produces the correct genotype
6. **Edge case generation:** Each `CorruptionMode` produces the expected defect
7. **Golden file stability:** Golden file content matches a snapshot (snapshot testing)
8. **Sex handling:** Male genomes have hemizygous X-linked loci, female genomes have diploid
9. **Population frequencies:** Generated genotype distributions roughly match expected HWE for the specified population (chi-squared goodness-of-fit over 10,000+ variants)

### 9.5 Relationship to Existing Tests

The factory should be adopted incrementally:
1. **Phase 1:** Replace inline test fixtures in `parser.test.ts` with factory-generated files
2. **Phase 2:** Use golden files for `carrier.test.ts`, `traits.test.ts`, `pgx.test.ts`, `prs.test.ts`
3. **Phase 3:** Use couple scenarios for integration tests (end-to-end analysis pipeline)
4. **Phase 4:** Use edge case files for robustness/regression testing

### 9.6 Known Parser Gaps Exposed by This Spec

The following parser issues will be discovered when running edge case tests. They are documented here so the implementer and the parser maintainer are aligned:

| Gap | Edge Case | Current Behavior | Recommended Fix |
|-----|-----------|------------------|-----------------|
| **BOM handling** | `bom` corruption mode on MyHeritage | BOM character prefixes first CSV field, causing header detection failure | Strip `\uFEFF` from first line in `parseGeneticFile()` or `detectFormat()` |
| **Bare CR line endings** | `mixed_line_endings` with `\r` only | Lines separated by bare `\r` are concatenated (not split by `indexOf('\n')`) | Add `\r`-only splitting to `iterateLines()` |
| **Hemizygous X-linked** | Golden 5, X-linked male scenarios | `determineCarrierStatus()` returns `'unknown'` for single-char genotypes | Add hemizygous detection: single allele matching pathogenic = affected |
| **VG prefix in VCF** | MyHeritage proprietary IDs | VCF parser only accepts `rs` prefix | Not applicable -- VG IDs are MyHeritage-only, not present in VCF files |
