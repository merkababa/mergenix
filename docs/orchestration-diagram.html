Section 1

Architecture Overview

The orchestration model uses five distinct layers. Each layer has a single responsibility. Code only flows downward; results flow upward.

Layer 1

Conductor

Top-level orchestrator. Spawns teams, tracks progress, updates status files. Never writes code. Context window is sacred -- only summaries flow in.
▼
Layer 2

Dev Team Leader

Spawned via TeamCreate. Orchestrates executor agents with file ownership boundaries. Never writes code. Can delegate to Gemini where appropriate.
▼
Layer 3

Executor Agents

Actual code writers. Each owns specific files to prevent merge conflicts. Pre-embedded with Technologist + Business checklists. Report results upward to Dev Team Leader.
▼ ▼
Layer 4

Gemini Review Coordinator

Spawns separate Gemini instances per reviewer role. Each gets full code + diff in interleaved format. Uses enhanced prompts with adversarial questions and anti-inflation grading.
▼
Layer 5

Claude Opus Reviewers

Independent Claude Opus agents per reviewer role. Own context, full tool access, can read files and trace code paths. Results go to Judge/Synthesis agent, not combined.
Section 2

Conductor Rules

The Conductor's context window is the most valuable resource in the system. Every byte must earn its place. Long sessions lead to context compaction, which causes lost memory and hallucination.

⚠ NEVER Do Directly

  • Write or edit code files (.ts, .tsx, .py, .js, .css)
  • Read large source files (over 50 lines)
  • Run tests or builds directly
  • Debug failing tests
  • Paste code blocks in responses
  • Combine 10 reviewers into 1 agent
  • Send Gemini only a diff (needs full files)
  • Spawn single agent to fix 6+ issues

✓ ALWAYS Do

  • Orchestrate agents -- spawn, assign, track, summarize
  • Summarize agent output in 2-3 lines max
  • Batch work into agents (1 agent with 5 steps > 5 tool calls)
  • Use memory files for decisions and patterns
  • Background long tasks (Gemini reviews, test runs, builds)
  • Update PROGRESS.md, PROJECT_STATUS.md, CLAUDE.md
  • Run git commands (commit, push, branch, PR via gh)

⚠ Red Flags (You're Doing It Wrong)

  • Reading a .ts / .tsx / .py file -- should delegate to explore agent
  • Writing code in Edit/Write call -- should delegate to executor
  • Tool output exceeds 100 lines -- should have delegated
  • Running vitest/pytest directly -- should delegate to agent
  • Spawning 3+ fire-and-forget Tasks -- should use TeamCreate
  • Context filling up fast -- doing too much yourself
  • Dev Team Leader writing code -- should delegate to executors

Planning Phase

Before execution begins, the Conductor spawns separate planning agents -- one per reviewer perspective. Each agent analyzes the phase from their domain. The Conductor then aggregates all perspectives into a unified plan for user approval.

Planning Flow

  • Conductor asks user which reviewer perspectives to include (default: all 10)
  • Separate planning agent spawned per perspective
  • Each agent analyzes: file structure, test strategy, security threats, UX requirements, etc.
  • Conductor aggregates into unified plan
  • User approves before any execution begins

Section 4

Execution Phase

The Conductor spawns a Dev Team Leader via TeamCreate. The Leader splits work across executor agents with strict file ownership boundaries to prevent merge conflicts. Every executor prompt includes pre-embedded checklists.

⚒ Executor Enhancement

Every executor agent prompt MUST include:

  • Technologist Checklist: asyncio.to_thread for blocking ops, useCallback, React.memo, hoisted constants, next/dynamic, no N+1 queries
  • Business Checklist: correct terminology (one-time purchase), backend tier gating, price validation, upgrade CTAs, user-friendly errors
  • Reference to docs/DEVELOPMENT_GOTCHAS.md

✋ Layer 0: Static Analysis Gate

Executors MUST pass all static checks before code goes to any reviewer:

  • ruff check -- Python linting
  • tsc --noEmit -- TypeScript type checking
  • vitest --run / pytest -- all tests passing
  • No build errors

Filters out ~30% of mechanical issues that would waste reviewer cycles.

File Ownership Boundaries

Each executor agent owns specific files. No two agents edit the same file. This prevents merge conflicts and ensures clear responsibility. If work touches shared files, the Dev Team Leader coordinates sequencing.

Section 5

Three-Layer Review Pipeline

Code passes through a rigorous three-layer review pipeline before any PR is created. Static analysis gates mechanical issues. Gemini provides broad sweep and triage. Claude Opus performs deep independent analysis.

Layer 0: Static Analysis Gate

  • ruff check -- Python lint (unused imports, style violations)
  • tsc --noEmit -- TypeScript types (exclude known pre-existing errors in client.ts:157 and demo-results.ts:268)
  • All tests passing (vitest --run / pytest)
  • Must pass BEFORE any LLM reviewer sees code
▼

◈ Stage 1: Gemini Reviews (Broad Sweep + Triage)

  • Coordinator spawns separate Gemini instances per reviewer role
  • Each gets full code + diff in interleaved format (diff first, then full file)
  • Enhanced prompts with adversarial questions, mandatory checklists, and grading calibration
  • UNCERTAINTY flags for issues Gemini is not confident about -- escalated to Claude
  • Second-pass "breaker mindset" instruction for adversarial re-review
  • Anti-inflation grading: A+ = zero issues, tied to severity not overall feel
  • Alternative: scripts/gemini-review.sh script instead of Coordinator agent
  • Iterate until all relevant Gemini reviewers give A+
▼

⚖ Judge / Synthesis Layer

Spawned after each review stage (Gemini or Claude). Model: sonnet (synthesis, not deep analysis).

  • Deduplicate -- same issue flagged by multiple reviewers merged into one
  • Resolve conflicts -- if Technologist and Architect disagree, Judge decides by impact
  • Prioritize -- rank all unique issues: CRITICAL > HIGH > MEDIUM > LOW
  • Final grade table -- | Reviewer | Grade | Issues Found | Unique After Dedup |
  • Fix manifest -- ordered list of issues grouped by file with clear instructions

In Phase 7, reviewers flagged ~40 total issues but only ~25 were unique after dedup.

▼

☈ Pipeline Overlap: Gemini to Claude

  • As each Gemini role reaches A+, immediately start corresponding Claude Opus reviewer
  • Do not wait for ALL Gemini roles to pass -- overlap for non-overlapping file reviewers
  • If a fix touches files an already-running Claude reviewer is analyzing, re-run that reviewer
  • Projected savings: ~30-40% review time reduction
▼

◆ Stage 2: Claude Opus Independent Reviews (Deep Dive)

  • Separate Claude Opus agent per reviewer role -- own context, fully independent
  • Receives Gemini's UNCERTAIN flags as priority review list
  • Full tool access: reads files, traces code paths, simulates runtime behavior
  • Results are NOT combined -- each goes to Judge/Synthesis agent
  • Iterate until all Claude reviewers give A+

Grades Table Format

Section 6

The 10 Reviewer Roles

Each reviewer operates as a separate, independent agent with their own context. They never share context or combine results -- all output flows through the Judge/Synthesis layer.

Reviewer 1

☷ Architect

System design, modularity, separation of concerns, scalability, design patterns, dependency management, type safety. Enforces layer purity and ORM anti-patterns. Gets ALL files.

Reviewer 2

☑ QA

Test coverage, edge cases, assertions, error handling, regression risk, logging and observability. Checks happy paths, negative paths, nulls, zeros, and boundary conditions.

Reviewer 3

⚛ Scientist

rsID accuracy, genetics correctness, citations, scientific methodology, data integrity. Validates that genetic predictions match published GWAS data and population frequencies.

Reviewer 4

⚙ Technologist

Performance, React/Next.js patterns, bundle size, memory, async correctness, modern practices. Catches event loop blocking (bcrypt, Stripe SDK), N+1 queries, missing code splitting.

Reviewer 5

♦ Business

Tier gating, conversion funnel, upgrade CTAs, copy quality, naming accuracy, market fit. Enforces one-time purchase terminology, backend price validation, demo conversion hooks.

Reviewer 6

☺ Designer

Accessibility (ARIA roles, sr-only text, aria-controls), responsive design, heading hierarchy, keyboard navigation, UX flow. Focuses on WCAG compliance and visual consistency.

Reviewer 7

⚠ Security Analyst

OWASP top 10, SQL/command injection, CSRF, token handling, encoding, secrets management, timing attacks. Constant-time comparisons, webhook signature verification, rate limiting.

Reviewer 8

✎ Code Reviewer

Readability, naming conventions, DRY/SOLID principles, style consistency, dead code elimination, import hygiene, function length, magic string elimination. Gets all changed code files.

Reviewer 9

⚖ Legal + Privacy

GDPR compliance, GINA (genetic data), data retention, consent flows, right to deletion, cookie consent, age verification (COPPA), data export (Article 20), encryption adequacy, cross-border transfers.

Reviewer 10

♥ Ethics / Bioethics

Population bias in genetic data, responsible result framing, emotional harm prevention, informed consent UX, eugenics guardrails, disclaimer adequacy, vulnerable populations, data purpose limitation.

Section 8

Fix Flow

After reviews identify issues, the fix flow routes work based on issue count. After fixes, only the reviewer roles that were below A+ are re-reviewed.

6+ Issues: Full Team

  • Conductor spawns Dev Team Leader
  • Leader creates executor team with file ownership
  • Executors implement fixes in parallel
  • QA agent verifies tests still pass
  • Report summary back to Conductor

1-5 Issues: Single Executor

  • Conductor spawns single executor agent
  • Agent implements all fixes + runs tests
  • Reports pass/fail back to Conductor
  • A single executor handles up to 5 related issues efficiently

After Fixes

  • Re-review only the reviewer roles that were below A+
  • Use the same stage (Gemini or Claude) that identified the issues
  • Iterate until all re-reviewed roles give A+
  • Never spawn a single agent to fix 6+ issues -- always use the team flow
Section 9

Feedback Loop (Gemini Calibration)

After every Claude Opus review cycle, the Conductor executes a mandatory feedback loop. This is what makes Gemini improve over time. Skipping it degrades accuracy.

◆
Claude finds issues Gemini missed -- categorize by pattern (async-blocking, naming, security...)
▼
✎
Append to docs/gemini-calibration.md -- cumulative learning across phases
▼
⚙
Include CALIBRATION section in next Gemini review prompt with confirmed misses and dismissed flags
▼
☑
Track: Confirmed Misses, Confirmed UNCERTAIN Flags, Dismissed UNCERTAIN Flags
▼
⚠
After 3+ repeated misses for a pattern, skip Gemini for that reviewer role -- go directly to Claude

Calibration Prompt Format

CALIBRATION FROM PREVIOUS REVIEWS: - You previously missed: [list from gemini-calibration.md] - Your UNCERTAIN flag on [X] was CONFIRMED -- increase sensitivity - Your UNCERTAIN flag on [Y] was DISMISSED -- reduce sensitivity - Common false positives to AVOID: [list]

Section 10

Gemini Delegation Rules

Gemini is called exclusively via Bash CLI. MCP tools are broken on Windows and must never be used.

✓ A-Tier (Always Use)

  • Code review (with full files + diff)
  • Design review and visual analysis
  • Documentation writing
  • Multi-file analysis (1M context window)
  • JSON validation
  • CSS/UI review

✗ D-Tier (Never Use)

  • Authentication / authorization code
  • Payment processing code
  • Git operations
  • Multi-step agentic workflows
  • Debugging / troubleshooting
  • Codebase search

CLI Usage

# Always specify model explicitly (default is weaker gemini-2.0-flash) gemini -p "Your prompt here" --model gemini-3-pro-preview 2>&1 # Long prompt via file cat <<'EOF' > /tmp/gemini-prompt.txt Your long prompt here... EOF gemini -p "" --model gemini-3-pro-preview < /tmp/gemini-prompt.txt 2>&1
  • Always use run_in_background: true -- responses take 10-30s
  • Always append 2>&1 -- CLI prints status to stderr
  • NEVER fall back to weaker models on rate limit (429) -- wait and retry (60s, 120s, 300s)
  • Only delegate code to Gemini if it can match or exceed Claude quality

Section 12

Severity Definitions

CRITICAL

Exploitable vulnerability, data loss, financial loss, system crash. Examples: SQL injection, event loop blocking on high-traffic endpoint, charging wrong amount.

HIGH

Incorrect behavior in primary feature, significant performance issue, architectural violation. Examples: N+1 on core endpoint, business logic in router, reflected XSS.

MEDIUM

Edge case bug, best practice lapse, confusing code. Examples: missing negative test, poor variable name, inefficient but non-blocking operation.

LOW

Stylistic issue, typo, micro-optimization. Examples: comment typo, slightly verbose code, unused import.

Section 13

Grading Rubric

Grades are tied to issue severity, not overall feel. Anti-inflation rule: A+ should be exceptionally rare. If you have not found at least one potential concern, you have not looked hard enough.

A+
ZERO issues found after thorough adversarial analysis. Exceptionally rare.
A
Only LOW severity issues. Minor stylistic or micro-optimization concerns.
A-
At most one MEDIUM issue. A single edge case or best practice lapse.
B+
Multiple MEDIUM issues or one HIGH issue. Needs targeted fixes.
B
One CRITICAL issue. Security vulnerability, data loss risk, or system crash potential.
<B
Multiple HIGH or CRITICAL issues. Significant rework required.
